# Concept: Deterministic Model Colors via Hash

## Проблема

Предыдущий подход: семантические цвета для veo/eleven/openai + массив из 3 fallback-цветов по индексу.

Дефекты:
1. При > 3 неизвестных моделей — fallback цвета повторяются
2. Цвет зависит от алфавитного индекса, нестабилен при изменении набора моделей
3. Легенда использовала хардкодированный rose для всех элементов

## Решение: djb2 hash → oklch hue (без семантической привязки)

Семантические цвета (veo = rose, eleven = sky и т.д.) были условностью дизайн-макетов. В реальном продукте важны только уникальность и стабильность цвета — конкретный hue не имеет значения.

```ts
function hashHue(str: string): number {
  let h = 5381;
  for (let i = 0; i < str.length; i++) h = ((h << 5) + h) ^ str.charCodeAt(i);
  return Math.abs(h) % 360;
}

export function modelColor(model: string): string {
  return `oklch(0.68 0.18 ${hashHue(model)})`;
}
```

## Свойства

- **Детерминированность:** одно и то же имя всегда даёт один и тот же цвет, независимо от порядка сортировки или других моделей в наборе
- **Распределение:** djb2 хорошо распределяет hue по кругу для типичных имён вида `gemini-2.5-flash`, `gemini-2.5-pro`, `imagen-3`
- **Читаемость:** oklch(0.68 0.18 hue) — достаточно светлый и насыщенный для тёмного фона
- **Нет зависимостей:** 2 строки кода
- **Нет `m.includes` эвристик:** не требует обновления при появлении новых провайдеров

## SVG Gradient IDs

Параллельный фикс: ID градиентов строятся из имени модели, не из индекса:

```ts
function gradId(model: string): string {
  return `burn-grad-${model.replace(/[^a-zA-Z0-9]/g, "-")}`;
}
```

Это исключает конфликты SVG IDs при двойном рендере (React StrictMode) или изменении порядка моделей.

## Экспорт

`modelColor` экспортируется из `GlobalCapitalBurnChart.tsx` и импортируется в `GlobalAnalyticsSection.tsx` для построения легенды — единый источник истины для всех цветов моделей.
